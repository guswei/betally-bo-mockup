給 Donald 的操作手冊 — 怎麼指揮弱模型過活

2026-07-03 Fable 5 寫。讀者是你本人，不是模型。制度最可靠的執行點是你的一句話。

1. 開場句式庫（照抄，換掉 <> 內容）

任務開場句
PRD / spec「先讀 .claude/rules/prd-standard.md 再動工：<需求>。完成後跑 scripts/check-prd.sh <檔> 把結果貼給我。」
Mockup「先讀 .claude/rules/mockup-annotation.md。做 <頁面>。完成標準：UI 英文/RD 註記繁中、欄位標型別與必填、瀏覽器點測過每個互動。」
改程式「IN SCOPE 只有 <檔案清單>，其他都不准動。改完 npm run build＋瀏覽器實測，貼證據。」
大範圍調查「派 Explore subagent 掃 <目錄> 回報結論＋檔案:行號，不要自己逐檔讀。」
交付收尾「派 verifier 審 <交付物路徑>，審查基準 <規則檔路徑>，修完再回報。」
查證事實「派 researcher 查證：<問題清單>。只收 CONFIRMED＋URL 或 NOT FOUND。」

2. 驗收條件公式（把需求說死，模型才不會飄）

公式：<動詞><對象>＋「完成的定義是 <可判定結果>」＋「證據是 <輸出/URL/路徑>」

❌ 模糊✅ 可判定
幫我看一下這個報表確認 DepositList 的日期篩選在跨月時是否漏資料，給我重現步驟或「確認無誤」的測試證據
把 mockup 弄好看一點統計卡改成 4 欄、間距對齊 SystemConfig 頁的風格，改完貼截圖
修那個 bug先寫出重現步驟，修完後同樣步驟不再重現，貼前後對照

3. 唬爛訊號（看到就追問，一句話戳破）

訊號追問
「已完成」但沒列檔案路徑「列出每個交付物的完整路徑。」
「測試/build 通過」但沒貼輸出「貼 build 輸出最後 10 行。」
「應該沒問題」「大致上」「基本上」「哪一項沒實際驗證？列出來。」
過程很快但回報一大串完成項「派 verifier read-back 這批交付。」
改 A 順便「優化」了 B「B 是誰叫你改的？回滾到只改 A。」
呼叫某 skill/agent 失敗後開始繞路「查 system prompt 清單，它存在嗎？不存在就換路。」

4. 什麼時候叫它升級 / 換路（你只要說一句）

  • 同一個錯修兩次還在錯 → 「停。帶完整失敗軌跡（做了什麼/輸出/為何錯）按 model-dispatch §4 升級。」
  • 每修一次冒一個新錯 → 「停。這是打地鼠，按 judgment-rubrics §4 換方法，不要再重試。」
  • 它開始跟工具/框架對著幹 → 「回到上一個能動的狀態，重想方案。」

5. 每月 10 分鐘維護

  1. 掃一眼 .claude/rules/lessons.md——有反覆出現的教訓就叫模型提案升格進規則檔（B 級，它會來問你）。
  2. 跑 ls ~/.claude/agents ~/.claude/skills，跟 skills-inventory.md 對一下，叫模型更新快照。
  3. usage 儀表板（claude.ai/settings/usage 或 /usage）看一次消耗；第一次順便驗證 subagent 是否計入額度，結果叫模型記進 lessons.md。

6. 只記一件事

弱模型＋明確驗收條件 > 強模型＋模糊指示。
你多花 30 秒把任務說死，省掉的是三輪返工。
