Iovation Report
本需求範圍:既有 Iovation Report(唯讀查核紀錄)新增兩欄 — Evidence(已報狀態)與 Actions(Add Evidence)。不動既有過濾區、既有欄位、Export CSV、device check 流程。evidence 為「事後回報真實結果給 iovation」,屬風控/不可逆操作。
既有過濾條件、欄位、分頁與 Export CSV 均維持原狀,本期不改。

Evidence 欄(NEW):顯示該列是否已回報過 evidence。iovation 無「查詢我送過哪些 evidence」的 API → 此欄資料一律讀我方自存的 evidence 紀錄表(見下方資料模型註記),不向 iovation 即時查。未報過顯示 —;已報顯示類型 badge(hover 可看時間/操作者)。
Actions 欄(NEW):每列 Add Evidence 按鈕 → 開 modal。權限:所有可進此 Iovation Report 的操作者皆可送出。⚠️ 風險註記:送出會回灌 iovation 且不可逆 → 故 modal 內強制二次確認當補償控制。日後若要收緊為「限管理層/獨立權限位」屬改版。

資料模型(我方新增表,給 RD) — iovation_evidence_log(審計鏈,金流/風控敏感):
• id PK · iovation_report_id FK(關聯被報的那列 check) · usercode(=iovation account code) · device_id(=device alias) · source_txn(原始 check 的 iovation txn,可空)
• evidence_type enum(🚩 合法值待確認,見下) · comment VARCHAR · idempotency_key UNIQUE(防重送)
• iovation_status(iovation 回傳) · iovation_evidence_id(回傳識別,可空) · operator_id · operator_role · created_at
審計欄(operator/時間/回傳結果)不可事後竄改;此表即「已報狀態」與稽核來源。

脈絡(唯讀,帶自該列):送出時後端用 usercode(account code)+ device_id(device alias)+ source_txn 綁定,讓 evidence 精準掛到「這次查核的這台裝置」。操作者不可改這些值。
🚩 Blocker — enum 待 iovation 確認:上列為常見假設值。實際合法 evidence_type 是 iovation 按你帳號在 portal 設定的,無動態 list API → 定工時前 RD 請翻 device check 那份整合文件確認實際開了哪些,再以此為準。必填。

操作者填的回報訊息,對應 iovation Evidence 的 comment/evidence value。VARCHAR(500)(上限待對 iovation 欄位上限);必填、去頭尾空白、不可全空白。純文字落地,顯示時須 escape 防 XSS。
不可逆 → 強制二次確認:未勾選不可送(補償控制,因權限選最寬鬆)。
Idempotency(防重送):送出帶 Idempotency-Key;同一 (report_id + type) 重複送出 → 後端去重,不重複打 iovation。注意 Iovation Report 本就有同一裝置多列重複紀錄,需避免同一查核被連點重報。
