Iovation Report
本需求範圍:既有 Iovation Report(唯讀查核紀錄)新增兩欄 — Evidence(已報狀態)與 Actions(Add Evidence / Retract-Ignore)。不動既有過濾區、既有欄位、Export CSV、device check 流程。evidence 為「事後回報真實結果給 iovation」,屬風控/不可逆操作。
既有過濾條件、欄位、分頁與 Export CSV 均維持原狀,本期不改。

Evidence 欄(NEW):顯示該列是否已回報過 evidence。iovation 無「查詢我送過哪些 evidence」的 API → 此欄資料一律讀我方自存的 evidence 紀錄表(見下方資料模型註記),不向 iovation 即時查。未報過顯示 —;已報顯示類型 badge(hover 看代碼/時間/操作者);已撤回顯示刪除線 Retracted / Ignored。
Actions 欄(NEW):每列 Add Evidence 按鈕 → 開新增 modal;已報過的列另出現 Retract / Ignore 按鈕 → 開撤回 modal。權限:所有可進此 Iovation Report 的操作者皆可送出/撤回。⚠️ 送出與撤回都會回灌 iovation 且不可逆 → 兩個 modal 內皆強制二次確認當補償控制。日後若要收緊為「限管理層/獨立權限位」屬改版。

資料模型(我方新增表,給 RD) — iovation_evidence_log(審計鏈,比照金流等級管控):
• id PK · iovation_report_id FK(關聯來源那列 check,做稽核回溯) · action_type enum(SUBMIT/RETRACT/IGNORE) · applied_to_type enum(ACCOUNT/DEVICE) · user_type enum(PLAYER/DEVICE)
• usercode(=iovation account code) · device_id(=device alias) · source_txn(原始 check 的 iovation txn,可空)
• evidence_type(iovation 代碼,如 1-1/3-4/100-1,見下方清單) · comment VARCHAR · idempotency_key UNIQUE(防重送)
• iovation_status(iovation 回傳) · iovation_evidence_id(回傳識別,可空) · operator_id · operator_role · created_at
審計欄(operator/時間/回傳結果)不可事後竄改;撤回/忽略另寫一筆 SUBMIT→RETRACT/IGNORE 紀錄,不覆蓋原 SUBMIT。此表即「已報狀態」與稽核來源。
🚩 待 RD 對照 iovation 整合文件確認(2 點):
①「Applied to type」(Account/Device) 與「User type」(Player/Device) 語意疑似重疊 — 客戶回饋把兩者列為獨立欄位,但 Device 值同時出現在兩個下拉。RD 請對照 device check 那份 iovation 整合文件,確認這是兩個不同 API 參數(如 target 掛載對象 vs 帳號屬性),還是同一參數的重複,再決定是否合併。mockup 先忠實照客戶兩個欄位呈現。
